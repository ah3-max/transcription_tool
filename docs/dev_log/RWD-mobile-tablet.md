# 開發日誌：前端 RWD — 平板斷點補齊與三層級重構

> 類型：開發日誌（事中／事後，逐步追加）
> 配對規劃：`docs/plan-log/RWD-mobile-tablet.md`
> 格式：原本怎麼樣 → 發生什麼 → 改了什麼 → 遇到什麼困難 → 又怎麼改 → 最後如何解決

---

**① 2026-06-26｜平板斷點補齊（單一 ≤860 拆成 ≤1024＋≤600 三層級）**
- 原本：`web/index.html`（單檔、CSS 內嵌）只有 2 層 RWD：`@media(max-width:860px)`（手機）＋其餘桌機，另有 `@media(max-width:460px){.ph}` 一條。iPad 直向 768–834px 落入手機層被過度精簡（資源量/即時原文欄/表格端點欄/波形全隱藏）；861–1024px 觸控平板卻吃桌機左側欄。
- 發生：盤點 iPhone Safari / Chrome 手機優化時，確認「缺中間平板層」。讀檔時另發現：高齡大字「愛愛院覆寫區」位於 media query **之後**，其無條件規則（`.stage-main{--cap:30px}`、`.ph{font-size:28px}`）以原始碼順序蓋掉同特異度的 media query 規則 → 手機 `--cap:21px` 與 `≤460 .ph:19px` 皆為**既有死碼**（非本次造成）。
- 改了什麼（使用者核准「補平板缺口」）：把 `@media(max-width:860px)` 拆成
  ① `@media(max-width:1024px)`（版面層）：`.side` 隱藏、`.app` 單欄、`.mnav` 底欄顯示、`.main` 留底欄淨空 92px；
  ② `@media(max-width:600px)`（精簡層，置於 1024 之後）：才隱藏資源量/原文欄/表格端點欄/波形/場次 meta/qrow 進度條，縮 padding 與 `--cap`、`.top` 收緊。
  並把原第 258 行 `@media(max-width:860px){.top-divider/#resToggle…}` 改為 `max-width:600px`。中間 **601–1024 自然成為「保留高資訊密度的平板層」**（底欄＋單欄，但資源量/原文欄可開/表格端點欄/波形/meta 全留）。
- 遇到困難：拆層後須保證「≤600 視窗所套用規則的聯集」≈ 原 ≤860 行為，否則手機會退化；且因 media query 在覆寫區之前，`.main`/`.top` 的 padding 需靠「後寫的 ≤600 區塊」蓋過「≤1024 區塊」。
- 又怎麼改：逐條核對聯集——`.main` padding 由 ≤600 的 `18px 16px 84px` 後寫覆蓋 ≤1024 的 `22px 26px 92px`、`.top` 同理；平板段**刻意不寫任何隱藏規則**以承襲桌機內容密度。`--cap` 維持與原碼相同處理（不動覆寫區，故手機仍沿用既有觀感，死碼留待另案）。
- 最後如何解決：`grep` 確認全檔已無 `860`，斷點為 `1024 / 600 / 600 / 460`，三層級成形。**靜態驗證（級聯推導＋grep）通過；尚未經真機／瀏覽器渲染實測。**
- 待續：① 真機或瀏覽器於 ~390 / ~800 / ~1280 三寬度渲染確認（底欄顯示、資源量「平板留・手機隱」、表格端點欄「平板留・手機隱」、無水平捲動、底欄不遮內容）；② `100vh → 100dvh`（`.app`，1 行，待使用者點頭）；③ 既有死碼 `--cap:21px`／`≤460 .ph:19px` 暫留並標記；④ 將「手機定位（收音交給電腦；手機做上傳/檢視/下載）」與 iOS Safari 檢核清單寫入 spec（建議 D-16，另案）。
